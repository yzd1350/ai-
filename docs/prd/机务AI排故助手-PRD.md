# 机务AI排故助手 — 产品需求文档

## 1. 项目概述

### 1.1 项目名称
**机务AI排故助手** (Aircraft Maintenance AI Troubleshooting Assistant)

### 1.2 一句话描述
帮助民航机务维修人员通过输入故障现象，快速获得结构化排故建议的 AI Web 应用。

### 1.3 目标用户
- **主要用户**：民航机务维修人员（一线排故工程师）
- **使用场景**：机坪、机库中遇到飞机故障，通过手机或电脑查询排故方案
- **用户痛点**：手册查阅耗时长、经验依赖度高、复杂故障排查步骤多且容易遗漏

### 1.4 项目性质
个人学习项目 + MVP 验证，当前阶段为单用户模式。

---

## 2. 产品目标

### 2.1 核心目标

| 目标 | 衡量标准 |
|------|----------|
| 输入故障现象 → 获取排故建议 | 从输入到展示结果 < 15 秒 |
| 检索相关手册知识 | 向量检索命中 top 3 相关片段 |
| 结构化排故输出 | 每次输出包含：可能原因、处理步骤、AMM 章节、工具航材 |
| 历史记录可回溯 | localStorage 持久化，刷新不丢失 |

### 2.2 非目标（MVP 不做）
- 多用户注册/登录系统
- 权限管理 / RBAC
- 实时协作
- 故障数据统计分析
- 移动端原生 App（使用响应式布局替代）

## 4. 功能需求（含完整分支流程）

### 4.1 F1 — 首页搜索 + 输入控制

**主流程**：输入故障现象 → 点击"开始排故" → 进入加载态

**分支流程**：

| 场景 | 触发条件 | 处理方式 |
|------|----------|----------|
| **空输入提交** | 输入框为空或纯空格 | 按钮置灰（disabled），不可点击 |
| **快捷键提交** | Ctrl + Enter | 行为与点击按钮完全一致 |
| **加载中重复提交** | 正在 loading 时再次提交 | 按钮和输入框同时 disabled，阻止重复提交 |
| **示例问题快捷填入** | 用户点击"试试这些问题"中的标签 | 自动填入输入框，不自动提交，允许用户修改后再提交 |
| **输入过长** | 超过 2000 字符 | 不阻止，但显示字数提示："已输入 xxx/2000 字"（建议后端做截断） |
| **页面刷新** | 结果页按 F5 刷新 | 回到首页搜索态（结果不持久化） |
| **浏览器后退** | 从结果页后退 | 回到首页搜索态 |
| **输入聚焦** | 点击输入框 | 输入框边框高亮（ring），提示文字消失 |
| **输入失焦** | 点击输入框外 | 边框恢复，保留已输入内容 |

### 4.2 F2 — 后端检索与 AI 分析（含异常分支）

**主流程**：
```
用户问题
  → OpenAI Embeddings 向量化
  → Supabase pgvector 相似度搜索（取 top 3）
  → 构造 Prompt：系统提示词 + 3 个知识片段 + 用户问题
  → DeepSeek V4 Pro 生成回答
  → 解析 JSON 返回前端
```

**异常分支**：

| 异常场景 | 触发条件 | 前端处理 | 后端响应 |
|----------|----------|----------|----------|
| **知识库为空** | Supabase documents 表无数据 | 显示专用提示："知识库尚未初始化，请先运行知识导入脚本" | `{ success: false, error: "KNOWLEDGE_BASE_EMPTY" }` |
| **检索无结果** | 向量搜索返回 0 条记录 | 降级处理：不注入知识片段，仅靠模型自身知识回答，并在回答顶部标注"未在知识库中找到直接相关条目" | `{ success: true, data: {...}, warning: "NO_MATCH" }` |
| **DeepSeek API 超时** | 超过 30 秒未响应 | 前端提示："AI 服务响应超时，请稍后重试"，后端做超时控制（30s） | `{ success: false, error: "AI_TIMEOUT" }` |
| **DeepSeek API 返回非 JSON** | 模型未按要求返回结构化 JSON | 后端做容错解析：用正则提取关键字段，失败则返回原文作为 summary | `{ success: true, data: { summary: rawText, ... }, warning: "PARSE_FALLBACK" }` |
| **DeepSeek API Key 无效** | 401 Unauthorized | 后端捕获并返回统一错误，前端提示："API Key 无效，请检查 server/.env 配置" | `{ success: false, error: "AUTH_FAILED" }` |
| **OpenAI Embeddings Key 无效** | 401 Unauthorized | 同上，提示："Embedding API Key 无效" | `{ success: false, error: "EMBEDDING_AUTH_FAILED" }` |
| **Supabase 连接失败** | 网络故障或配置错误 | 前端提示："知识库连接失败，请检查 Supabase 配置" | `{ success: false, error: "DB_CONNECTION_FAILED" }` |
| **网络中断（前端→后端）** | fetch 抛出异常 | 前端 catch 后显示："网络连接失败，请检查后端服务是否启动" + 回到搜索态 | 不涉及 |
| **请求体为空** | POST body 缺少 question 字段 | 后端校验，返回 400 | `{ success: false, error: "INVALID_REQUEST" }` |

### 4.4 F4 — 历史记录（增删改查完整分支）

**数据结构**：
```js
{
  id: "lq3kf8x2",        // 唯一标识（时间戳+随机数）
  question: "发动机启动时...", // 用户输入的完整问题
  summary: "该故障表现为...",  // AI 返回的摘要（前 100 字）
  createdAt: 1716379200000,   // 创建时间戳（毫秒）
}
```

**CRUD 操作矩阵**：

| 操作 | 触发方式 | 前置条件 | 行为 | 结果 |
|------|----------|----------|------|------|
| **创建 (C)** | 每次成功提交问题 → 获得结果后 | 有有效结果 | 自动将问题+摘要写入 localStorage，插入列表顶部 | 侧边栏历史卡片显示计数 +1 |
| **查看 (R)** | 点击侧边栏"历史记录"卡片 | 卡片已展开 | 展开历史记录列表；首次打开默认折叠 | 历史列表面板展开 |
| **回看详情 (R)** | 点击历史列表中的某条记录 | 该记录存在 | 关闭侧边栏 → 以该问题重新查询 AI → 展示新结果 | 进入结果页 |
| **删除单条 (D)** | hover 某条记录 → 点击 ✕ 按钮 | 该记录存在 | 从 localStorage 中移除该条；列表即时刷新 | 计数 -1 |
| **清空全部 (D)** | 点击"清空全部"按钮 | 列表非空 | 弹出确认提示 → 清空 localStorage 中所有历史 | 计数归零，显示空状态 |
| **空状态** | 无历史记录 | 列表为空 | 显示"暂无历史记录，输入故障现象开始排故" | — |
| **存储上限** | 超过 50 条 | — | 自动删除最旧的记录，保持最多 50 条 | 不阻塞创建 |

**历史记录与页面状态联动**：

| 场景 | 行为 |
|------|------|
| 在结果页点击历史记录 | 不回到搜索页，直接以历史问题重新查询 |
| 在加载态点击历史记录 | 取消当前加载，以历史问题重新查询 |
| 在错误态点击历史记录 | 关闭错误提示，以历史问题重新查询 |
| 当前问题与历史问题相同 | 仍然重新查询（用户可能想获取最新结果） |

### 4.5 F5 — 知识库导入脚本

**主流程**：
```
遍历 knowledge/ 下所有 .txt/.md/.pdf 文件
  → 读取文件内容
  → RecursiveCharacterTextSplitter 分段 (chunk_size=500, chunk_overlap=50)
  → 每段调用 OpenAI Embeddings 生成向量
  → 存入 Supabase documents 表
  → 输出导入统计
```

**分支流程**：

| 场景 | 处理方式 |
|------|----------|
| knowledge/ 目录为空 | 输出："未找到任何文档文件，请在 knowledge/ 目录下放入 .txt 或 .md 文件" |
| 文件编码非 UTF-8 | 尝试自动检测编码并转换；失败则跳过并输出警告 |
| 单文件超过 10MB | 输出警告，跳过该文件（避免内存溢出） |
| Supabase 连接失败 | 输出错误详情并终止脚本 |
| 某段 Embedding 生成失败 | 重试 2 次；仍失败则跳过该段，输出警告，继续处理后续段落 |
| 重复运行脚本 | 每次运行前弹出确认："检测到 documents 表已有数据，是否清空后重新导入？(y/n)" |
| 导入成功 | 输出："成功导入 X 个文件，共 Y 个知识片段" |
| 部分成功 | 输出："导入完成：X 个文件成功，Y 个片段已入库；Z 个片段跳过（详见日志）" |

## 6. 错误码与提示文案规范

| 错误码 | 前端提示文案 | 出现场景 |
|--------|-------------|----------|
| `KNOWLEDGE_BASE_EMPTY` | "知识库尚未初始化，请先运行知识导入脚本。参考 README.md 第 5 节。" | documents 表为空 |
| `NO_MATCH` | （非错误，结果正常展示，顶部灰字提示："未在知识库中找到直接相关条目，以下分析基于通用机务知识。"） | 检索得分为 0 或低于阈值 |

| `PARSE_FALLBACK` | （非错误，结果正常展示，但 structured 标记为 false） | AI 未返回合法 JSON |
| `AUTH_FAILED` | "API Key 无效，请检查 server/.env 配置。" | 401 |
| `EMBEDDING_AUTH_FAILED` | "Embedding API Key 无效，请检查 server/.env 配置。" | 401 |
| `DB_CONNECTION_FAILED` | "知识库连接失败，请检查网络和 Supabase 配置。" | 网络故障 |
| `INVALID_REQUEST` | "请求参数不合法，请重新输入故障现象。" | question 为空或缺失 |
| `NETWORK_ERROR` | "网络连接失败，请检查后端服务是否已启动（npm run dev）。" | fetch 异常 |
| `SERVER_ERROR` | "服务器内部错误，请稍后重试。" | 500 或其他未知错误 |

## 9. 数据生命周期

```
输入问题
  │
  ├─→ localStorage: 仅存储历史记录（问题+摘要+时间戳）
  │     ├─ 创建：每次成功排故后
  │     ├─ 删除：用户手动删除或达到 50 条上限
  │     └─ 持久化：浏览器 localStorage，不清缓存则永久保留
  │
  └─→ 内存（React State）: 当前页面状态
        ├─ 当前问题、当前结果、加载状态
        ├─ 页面刷新后丢失
        └─ 不写入任何持久化存储
```

## 10. 技术架构

### 10.1 技术栈

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端框架 | React 19 + Vite | 快速开发、HMR 热更新 |
| 样式方案 | Tailwind CSS 4 | 移动端优先、原子化 CSS |
| 后端框架 | Express.js 5 | 轻量级 HTTP 服务器 |
| 向量数据库 | Supabase + pgvector | 免费额度够用，托管免运维 |

## 11. 数据库设计

### 11.1 Supabase `documents` 表

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE documents (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  content TEXT NOT NULL,
  embedding VECTOR(1536),
  metadata JSONB DEFAULT '{}'::jsonb,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX ON documents
  USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

CREATE OR REPLACE FUNCTION match_documents(
  query_embedding VECTOR(1536),
  match_count INT DEFAULT 3
)
RETURNS TABLE(
  id UUID,
  content TEXT,
  metadata JSONB,
  similarity FLOAT
)
LANGUAGE plpgsql
AS $$
BEGIN
  RETURN QUERY
  SELECT
    documents.id,
    documents.content,
    documents.metadata,
    1 - (documents.embedding <=> query_embedding) AS similarity
  FROM documents
  ORDER BY documents.embedding <=> query_embedding
  LIMIT match_count;
END;
$$;
```
